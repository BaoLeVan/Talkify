# 05 — Giải pháp cho các vấn đề thực tế

## 5.1 Message Ordering — Đảm bảo thứ tự tin nhắn

### Snowflake ID (64-bit)

```
┌──────────┬─────────────────┬──────────────┬────────────────┐
│  1 bit   │    41 bits      │   10 bits    │   12 bits      │
│ unused   │ timestamp (ms)  │ machine ID   │ sequence/ms    │
│          │ since epoch     │ (1024 nodes) │ (4096 IDs/ms)  │
└──────────┴─────────────────┴──────────────┴────────────────┘

Epoch: 2024-01-01T00:00:00Z
→ Globally unique + time-sortable
→ ~69 years từ epoch
→ Extract timestamp từ ID → không cần stored createdAt riêng
```

### Sequence Number per Conversation (Redis INCR)

| ID | Mục đích | Đặc điểm |
|----|----------|----------|
| **Snowflake ID** | Global unique identifier, message lookup | Time-sortable, dedup |
| **Sequence Number** | Ordering trong conversation | Strict monotonic, Redis ATOMIC |

```java
// Redis INCR đảm bảo atomicity across multiple instances
public long nextSequence(String conversationId) {
    return redisTemplate.opsForValue().increment("seq:" + conversationId);
}
```

### Client-side Gap Detection

```javascript
// Phát hiện gap → request missing messages
for (let i = 1; i < messages.length; i++) {
    const gap = messages[i].sequenceNumber - messages[i-1].sequenceNumber;
    if (gap > 1) {
        fetchMessages(convId, messages[i-1].sequenceNumber, messages[i].sequenceNumber);
    }
}
// Sort trước khi render
messages.sort((a, b) => a.sequenceNumber - b.sequenceNumber);
```

## 5.2 Reliability — At-Least-Once Delivery

### 3-Phase ACK Mechanism

```
┌──────────────────────────────────────────────────────────────────────┐
│  Phase 1: CLIENT → SERVER (Message Sent)                             │
│  ─────────────────────────────────────────                           │
│  Client gửi msg với clientMessageId (UUID)                           │
│  Server xử lý → trả ACK {status: "SENT", serverMessageId}           │
│                                                                      │
│  ⚠️ Không nhận ACK trong 5s → RETRY với cùng clientMessageId       │
│  ✅ Server dùng clientMessageId làm idempotency key (Redis, TTL 24h) │
├──────────────────────────────────────────────────────────────────────┤
│  Phase 2: SERVER → RECIPIENT (Message Delivered)                     │
│  ─────────────────────────────────────────                           │
│  Server push NEW_MESSAGE → recipient WebSocket                       │
│  Recipient gửi DELIVERY_ACK → Server                                │
│  Server push MESSAGE_DELIVERED → sender                              │
│                                                                      │
│  ⚠️ User offline hoặc không ACK trong 30s:                         │
│     → Lưu pending_messages (MongoDB TTL 30 ngày)                    │
│     → Send Push Notification (FCM/APNs)                             │
│     → Khi user online: sync pending messages                        │
├──────────────────────────────────────────────────────────────────────┤
│  Phase 3: RECIPIENT confirms READ                                    │
│  ─────────────────────────────────────────                           │
│  Recipient mở conversation → gửi READ {lastReadSequence}            │
│  Server push MESSAGE_READ → sender                                   │
└──────────────────────────────────────────────────────────────────────┘
```

### Idempotency Implementation

```java
public MessageDTO sendMessage(SendMessageCommand cmd) {
    // 1. Check idempotency
    String dedupKey = "dedup:msg:" + cmd.clientMessageId();
    String existingId = redis.opsForValue().get(dedupKey);
    if (existingId != null) {
        return messageRepository.findById(Long.parseLong(existingId))
            .map(this::toDTO).orElseThrow();
    }

    // 2. Validate & Create
    Conversation conv = conversationRepository.findById(cmd.conversationId())
        .orElseThrow(() -> new NotFoundException("Conversation not found"));
    conv.validateParticipant(cmd.senderId());

    // 3. Atomic sequence number
    long seqNum = messageOrderingService.nextSequence(cmd.conversationId());

    // 4. Create domain object
    Message message = Message.create(conv.getId(), cmd.senderId(),
                                     cmd.content(), cmd.type(), seqNum);

    // 5. Save to MongoDB
    messageRepository.save(message);

    // 6. Update PostgreSQL conversation metadata
    conv.updateLastMessage(message.getId(), message.getContent().preview(100),
                           message.getCreatedAt());
    conversationRepository.save(conv);

    // 7. Store idempotency key
    redis.opsForValue().set(dedupKey, message.getId().toString(),
                            Duration.ofHours(24));

    // 8. Publish to Kafka for async delivery
    kafkaProducer.sendMessage(message);

    // 9. Publish domain events (in-process)
    message.getDomainEvents().forEach(eventPublisher::publishEvent);

    return toDTO(message);
}
```

### Offline Sync Flow

```java
public SyncResponse syncMessages(Long userId, long since, int limit) {
    // 1. Lấy conversations user tham gia
    List<ConversationParticipant> participations =
        participantRepository.findActiveByUserId(userId);

    SyncResponse response = new SyncResponse();

    for (var p : participations) {
        // 2. Lấy messages mới hơn lastReadSequence
        List<Message> newMsgs = messageRepository.findAfterSequence(
            p.getConversationId(), p.getLastReadSequence(), limit
        );
        if (!newMsgs.isEmpty()) {
            response.addConversation(p.getConversationId(), newMsgs,
                                     p.getUnreadCount());
        }
    }

    // 3. Xóa pending_messages đã sync
    pendingMessageRepo.deleteByUserId(userId);

    return response;
}
```

## 5.3 Performance — Thundering Herd (Group Chat lớn)

### Vấn đề
1 tin nhắn trong group 10,000 thành viên:
- 5,000 users online → 5,000 WebSocket pushes đồng thời.
- Write amplification cho unread counters nếu update ngay lập tức.
- Push notification storm cho 5,000 users offline.

### Giải pháp: Hybrid Fan-out Strategy

```
┌──────────────────────────────────────────────────────────────────────┐
│  Small Group (≤ 100 members): Fan-out On Write                       │
│  → Push realtime tới tất cả members ngay lập tức                    │
│  → Update unread counter cho tất cả                                  │
│  → Push notification cho offline members                             │
├──────────────────────────────────────────────────────────────────────┤
│  Large Group (> 100 members): Hybrid Fan-out                         │
│                                                                      │
│  1. Write message 1 lần vào MongoDB ✅                               │
│                                                                      │
│  2. Realtime: Batched Fan-out qua Kafka                              │
│     → Chia member list thành batches (50 users/batch)               │
│     → Kafka topic: "chat.fanout.{convId}"                           │
│     → 8 consumer threads xử lý song song                            │
│                                                                      │
│  3. Unread Counter: Fan-out On Read (lazy)                           │
│     → KHÔNG update ngay cho 10K users                               │
│     → Chỉ lưu conversation.sequenceCounter (1 write)                │
│     → unread = currentSeq - participant.lastReadSeq                  │
│     → Tính khi user mở app (O(1) query)                             │
│                                                                      │
│  4. Push Notification: Batched + Throttled                           │
│     → Chỉ gửi push cho users offline > 5 phút                      │
│     → FCM multicast (max 500 tokens/request)                        │
│     → Rate: tối đa 1 push/conversation/user/phút                    │
└──────────────────────────────────────────────────────────────────────┘
```

### Batched Fan-out Implementation

```java
@KafkaListener(topics = "chat.messages", groupId = "fanout-consumer", concurrency = "8")
public void processFanout(Message message) {
    List<Long> memberIds = participantRepository
        .findActiveMemberIds(message.getConversationId().value());
    memberIds.remove(message.getSenderId().value());

    if (memberIds.size() <= SMALL_GROUP_THRESHOLD) {
        // Direct delivery
        deliverToUsers(message, memberIds);
    } else {
        // Batched delivery
        Lists.partition(memberIds, BATCH_SIZE).forEach(batch ->
            kafkaTemplate.send("chat.fanout",
                message.getConversationId().value(),
                new FanoutBatch(message.getId().value(),
                                message.getConversationId().value(), batch))
        );
    }
}

private void deliverToUsers(Message msg, List<Long> userIds) {
    userIds.forEach(userId -> {
        String session = redis.opsForHash()
            .get("ws:sessions", userId.toString()).toString();
        if (session != null) {
            // Online → Redis Pub/Sub
            String instanceId = session.split(":")[0];
            redis.convertAndSend("ws:delivery:" + instanceId,
                buildWsPayload(msg, userId));
        } else {
            // Offline → Push notification (throttled)
            throttledPushService.enqueue(userId, msg);
        }
    });
}
```

### Push Notification Throttling

```java
public void enqueue(Long userId, Message msg) {
    String throttleKey = "push:throttle:" + userId + ":" + msg.getConversationId();

    // Chỉ gửi nếu chưa có push trong 60s
    Boolean isNew = redis.opsForValue()
        .setIfAbsent(throttleKey, "1", Duration.ofSeconds(60));

    if (Boolean.TRUE.equals(isNew)) {
        scheduler.schedule(() -> {
            int unread = getUnreadCount(userId, msg.getConversationId().value());
            sendFcmPush(userId, FcmPayload.builder()
                .title(msg.getConversationTitle())
                .body(unread > 1 ? unread + " tin nhắn mới" : msg.getContent().preview(80))
                .data(Map.of("conversationId", msg.getConversationId().value()))
                .build());
        }, 3, TimeUnit.SECONDS);
    }
}
```

## 5.4 Read Receipt Optimization

```java
public void markRead(String convId, Long userId, long lastReadSeq) {
    // 1. Update user's read position
    participantRepository.updateLastReadSequence(convId, userId, lastReadSeq);

    // 2. Tính unread count = 0
    redis.opsForValue().set("unread:" + userId + ":" + convId, "0");

    // 3. Notification per conversation type
    Conversation conv = convRepository.findById(convId).orElseThrow();
    int memberCount = conv.getMemberCount();

    if (conv.getType() == ConversationType.DIRECT || memberCount <= 20) {
        // Gửi chi tiết read receipt tới sender
        broadcastReadReceipt(convId, userId, lastReadSeq);
    } else {
        // Large group: chỉ track count qua HyperLogLog
        redis.opsForHyperLogLog().add(
            "read:" + convId + ":" + lastReadSeq, userId.toString()
        );
        // Client query "X người đã xem" khi hover vào tin nhắn
    }
}

public long getReadCount(String convId, long sequenceNumber) {
    return redis.opsForHyperLogLog().size("read:" + convId + ":" + sequenceNumber);
}
```

## 5.5 Rate Limiting

```java
@Component
public class MessageRateLimiter {

    // Sliding window: 5 msgs/sec, 60 msgs/min
    public void check(Long userId) {
        String secKey = "rl:" + userId + ":sec";
        String minKey = "rl:" + userId + ":min";

        Long secCount = redis.opsForValue().increment(secKey);
        if (secCount == 1) redis.expire(secKey, Duration.ofSeconds(1));
        if (secCount > 5) throw new RateLimitException("Gửi quá nhanh (5/giây)");

        Long minCount = redis.opsForValue().increment(minKey);
        if (minCount == 1) redis.expire(minKey, Duration.ofMinutes(1));
        if (minCount > 60) throw new RateLimitException("Đã đạt giới hạn 60 tin/phút");
    }
}
```
