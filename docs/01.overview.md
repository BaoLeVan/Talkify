## 1.2 Scaling WebSocket quy mô lớn

### Vấn đề cốt lõi
WebSocket là kết nối **stateful** — client duy trì TCP connection liên tục với **một instance cụ thể**. Khi scale horizontally, tin nhắn từ User A (Instance 1) phải đến được User B (Instance 3).

### Giải pháp: Redis Pub/Sub làm Message Bus

```
User A ──WSS──▶ Instance 1                       Instance 3 ──WSS──▶ User B
                    │                                  ▲
                    │ PUBLISH "ws:delivery:inst-3"     │ SUBSCRIBE
                    ▼                                  │
              ┌────────────────────────────────────────┐
              │           Redis Pub/Sub Cluster         │
              │   Channel: ws:delivery:{instanceId}    │
              └────────────────────────────────────────┘
```

**Session Registry (Redis Hash):**
```
HSET ws:sessions {userId} "{instanceId}:{sessionId}"
EXPIRE → cleaned on disconnect
```

**Cross-Instance Delivery:**
1. Instance 1 nhận tin nhắn từ User A.
2. Tra cứu Redis: User B đang ở Instance 3.
3. `PUBLISH ws:delivery:inst-3 <messagePayload>`.
4. Instance 3 lắng nghe channel của mình → push WebSocket tới User B.

### Capacity Planning
| Cấu hình Instance | Concurrent WebSocket Connections |
|------------------|----------------------------------|
| Spring MVC (Tomcat, 8GB RAM) | 50,000 – 80,000 |
| Spring WebFlux (Netty, 8GB RAM) | 100,000 – 150,000 |
| 10 instances (WebFlux) | ~1,000,000+ |

### Heartbeat & Auto-Reconnect
```
Client ──PING (30s)──▶ Server
Server ──PONG──────▶ Client

# Nếu không nhận PING trong 90s → Server đóng connection, cleanup session
# Client tự reconnect với Exponential Backoff: 1s → 2s → 4s → 8s → max 30s
```

## 1.3 Message Pipeline end-to-end

```
① Client A gửi WebSocket SEND
            ↓
② Server validate + Snowflake ID + SeqNumber (Redis INCR)
            ↓
③ ACK_SENT → Client A (ngay lập tức)
            ↓
④ Kafka PRODUCE → topic "chat.messages" (key = conversationId)
            ↓
    ┌────────────────────────────────────────────────┐
    │                                                │
    ▼                                                ▼
⑤a Kafka Consumer: Persist               ⑤b Kafka Consumer: Deliver
   → Write MongoDB (messages)                → Redis Pub/Sub lookup
   → Update PostgreSQL                       → WebSocket push
     (last_message, unread)                  → ACK_DELIVERED → Client A
            ↓
    User B offline?
            ↓
⑥ Push Notification (FCM/APNs)
   + Lưu pending_messages (MongoDB TTL 30d)
```

## 1.4 Load Balancing cho WebSocket

```nginx
# HAProxy Layer 4 — Least Connections + Sticky by UserId
backend websocket_backend
    mode tcp
    balance leastconn
    
    # Sticky session: cùng user → cùng instance
    stick-table type string len 64 size 1m expire 24h
    stick on req.hdr(X-User-Id)
    
    # Health check
    option httpchk GET /actuator/health
    
    server ws1 10.0.1.1:8080 check inter 5s
    server ws2 10.0.1.2:8080 check inter 5s
    server ws3 10.0.1.3:8080 check inter 5s
```

**Chiến lược:**
- **Least Connections** → instance ít connection nhất nhận connection mới.
- **Sticky Session by UserId** → cùng user về cùng instance, giảm cross-instance delivery.
- **Graceful Shutdown** → instance gửi `GOAWAY` → client reconnect sang instance khác.
- **Health Check** → tự động loại instance unhealthy.
