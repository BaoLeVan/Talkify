# 06 — Lộ trình Triển khai (Implementation Checklist)

## Phase 1 — Foundation & Infrastructure (Tuần 1–2)

```
☐ 1.1  Khởi tạo multi-module Maven project
       Modules: bootstrap, common, identity, messaging, group, media, notification, presence

☐ 1.2  Docker Compose environment
       Services: PostgreSQL, MongoDB, Redis, Kafka, MinIO, Prometheus, Grafana

☐ 1.3  Shared Kernel (chat-common)
       - AggregateRoot, BaseEntity, DomainEvent, ValueObject interfaces
       - SnowflakeIdGenerator (machine ID từ environment variable)
       - GlobalExceptionHandler + ErrorCode enum
       - ApiResponse<T> wrapper
       - Common DTOs (PageResponse, CursorPage)

☐ 1.4  CI/CD Pipeline (GitHub Actions)
       - Build → Test → Docker Build → Push → Deploy Staging

☐ 1.5  Database Migration Tool
       - Flyway cho PostgreSQL
       - MongoDB init script qua Docker entrypoint

☐ 1.6  Monitoring Stack
       - Prometheus scrape config
       - Grafana dashboards: JVM, Kafka, MongoDB, Redis
       - OpenTelemetry agent + Jaeger tracing
```

## Phase 2 — Identity & Authentication (Tuần 2–3)

```
☐ 2.1  User domain model (User, UserProfile, Device aggregate)

☐ 2.2  JWT Authentication
       - Access Token (15 phút) + Refresh Token (30 ngày)
       - Token rotation (Refresh Token one-time use)
       - Blacklist invalidated tokens in Redis

☐ 2.3  REST APIs
       - POST /auth/register, /auth/login, /auth/refresh, /auth/logout
       - GET/PUT /users/me, GET /users/{id}, GET /users/search

☐ 2.4  Device Registration API (FCM token management)

☐ 2.5  Spring Security configuration
       - JWT filter chain
       - CORS policy
       - Password encoding (BCrypt, cost=12)

☐ 2.6  Unit Tests + Integration Tests (≥ 80% coverage cho Identity module)
```

## Phase 3 — Core Messaging 1:1 (Tuần 3–5)

```
☐ 3.1  Conversation domain model + PostgreSQL schema

☐ 3.2  Message domain model + MongoDB collection + indexes

☐ 3.3  SnowflakeIdGenerator (thread-safe, configurable machine ID)

☐ 3.4  MessageOrderingService (Redis INCR per conversationId)

☐ 3.5  REST APIs
       - POST /conversations (tạo DIRECT)
       - GET /conversations (paged list)
       - GET /conversations/{id}/messages (cursor pagination)
       - POST /conversations/{id}/messages/read

☐ 3.6  WebSocket Configuration
       - STOMP over SockJS
       - JWT Handshake Interceptor
       - WebSocketSessionManager (Redis session registry)

☐ 3.7  Send Message Flow
       a. WebSocket handler nhận SEND event
       b. Validate + Rate limit
       c. Assign Snowflake ID + Sequence Number
       d. Return ACK_SENT ngay lập tức
       e. Kafka PRODUCE → "chat.messages"

☐ 3.8  Kafka Consumer — Message Persistence
       - Write MongoDB
       - Update PostgreSQL (last_message, sequence_counter)

☐ 3.9  Kafka Consumer — Real-time Delivery
       - Check Redis session (online/offline)
       - Online: Redis Pub/Sub → cross-instance WebSocket push
       - Offline: lưu pending_messages + push notification

☐ 3.10 3-Phase ACK (SENT → DELIVERED → READ)

☐ 3.11 Idempotency (clientMessageId dedup in Redis TTL 24h)

☐ 3.12 Read Receipts (WebSocket event + REST API)

☐ 3.13 Typing Indicator (Redis TTL 5s + WebSocket broadcast)

☐ 3.14 Integration Tests: end-to-end send/receive message
☐ 3.15 Load Test: 1,000 concurrent WebSocket connections
```

## Phase 4 — Group Chat (Tuần 5–7)

```
☐ 4.1  Group domain model (Group, GroupMember, GroupSettings aggregate)

☐ 4.2  REST APIs
       - CRUD Group (POST, GET, PUT, DELETE)
       - Participant management (add, remove, change role)
       - Invite link (create, revoke, join)
       - Group settings (mute, only-admins-send, max-members)

☐ 4.3  Role-based Permissions (OWNER > ADMIN > MEMBER)
       - Annotation-based permission check (@RequireGroupRole)

☐ 4.4  Group Fan-out Strategy
       - Small (≤ 100): direct WebSocket push
       - Large (> 100): Kafka batched fan-out (50 users/batch)

☐ 4.5  Unread Counter — Fan-out On Read
       - Tính từ: currentSeq - participant.lastReadSeq

☐ 4.6  Read Receipt Optimization
       - ≤ 20 members: individual receipts
       - > 20 members: HyperLogLog count-only

☐ 4.7  System Messages (member joined, left, role changed, group renamed)

☐ 4.8  Group Invitation Workflow (gửi, chấp nhận, từ chối, hết hạn)

☐ 4.9  Integration Tests cho group chat flows
```

## Phase 5 — Media & File Sharing (Tuần 7–8)

```
☐ 5.1  Media domain model + S3/MinIO storage adapter

☐ 5.2  File Upload APIs
       - Multipart upload (≤ 50MB)
       - Presigned URL cho file lớn (direct to S3)

☐ 5.3  Image Processing
       - Thumbnail generation (Thumbnailator library)
       - Resize theo breakpoints (800px, 400px, 200px)

☐ 5.4  MIME type validation + file size limits
       - Images: jpg, png, gif, webp (max 20MB)
       - Videos: mp4, mov (max 100MB)
       - Files: pdf, doc, zip,... (max 50MB)

☐ 5.5  Messages with attachments (IMAGE, FILE, AUDIO, VIDEO types)

☐ 5.6  CDN integration (Nginx/CloudFront caching headers)
```

## Phase 6 — Presence & Notifications (Tuần 8–9)

```
☐ 6.1  Presence Tracking
       - ONLINE khi connect WebSocket (HSET presence:{userId})
       - TTL 300s, renew mỗi heartbeat
       - OFFLINE khi TTL expire hoặc disconnect

☐ 6.2  Presence Broadcast
       - Broadcast tới contacts khi status thay đổi
       - Last seen tracking

☐ 6.3  FCM Integration (Firebase Admin SDK)
       - Multicast push (≤ 500 tokens/request)
       - Retry với exponential backoff
       - Token cleanup khi FCM trả lỗi "UNREGISTERED"

☐ 6.4  APNs Integration (iOS push)

☐ 6.5  Push Notification Throttling
       - 1 push/conversation/user/phút
       - Debounce 3s để gom messages

☐ 6.6  Notification Preferences
       - Mute per conversation (tắt trong 1h/8h/forever)
       - Global do-not-disturb setting
```

## Phase 7 — Offline Sync & Reliability (Tuần 9–10)

```
☐ 7.1  Pending Message Queue (MongoDB TTL index 30 ngày)

☐ 7.2  Sync API: GET /sync?since=&limit=
       - Delta sync: conversations changed + new messages

☐ 7.3  Client Reconnect Flow
       - Exponential backoff: 1s→2s→4s→8s→max 30s
       - Gửi sync request khi reconnect thành công

☐ 7.4  Message Gap Detection & Recovery
       - Client phát hiện gap trong sequenceNumber
       - Request missing messages: GET /conversations/{id}/messages?between=

☐ 7.5  Conversation List Sync (incremental by lastMessageAt)

☐ 7.6  Read Receipt Sync khi online lại
```

## Phase 8 — Optimization & Hardening (Tuần 10–12)

```
☐ 8.1  Rate Limiting (per-user, per-conversation)
       - Sliding window: 5 msgs/sec, 60 msgs/min

☐ 8.2  Message Search (Elasticsearch)
       - Index message text
       - Search by keyword, sender, date range

☐ 8.3  Database Optimization
       - MongoDB: explain() cho tất cả queries, compound indexes
       - PostgreSQL: EXPLAIN ANALYZE, connection pooling (HikariCP)

☐ 8.4  Redis Tuning
       - Eviction policy: allkeys-lru
       - Memory limit theo capacity plan

☐ 8.5  Kafka Tuning
       - 32 partitions cho "chat.messages"
       - Batch size, linger.ms cho producer
       - max.poll.records cho consumer

☐ 8.6  Load Testing (k6 / JMeter)
       - Target: 10K concurrent users, 50K messages/sec
       - Latency P99 < 200ms (send → deliver)

☐ 8.7  Security Hardening
       - SQL/NoSQL injection prevention
       - WebSocket message size limit (64KB)
       - JWT token rotation
       - OWASP checklist

☐ 8.8  API Documentation (Springdoc OpenAPI 3)
```

## Phase 9 — Production Readiness (Tuần 12–14)

```
☐ 9.1  Blue-green deployment (Docker Swarm / Kubernetes)

☐ 9.2  Database Backup
       - PostgreSQL: daily pg_dump + WAL archiving
       - MongoDB: daily mongodump + oplog backup

☐ 9.3  Centralized Logging (Log4j2 + Elasticsearch + Kibana + Filebeat)
              - Log4j2 AsyncAppender + EcsLayout → JSON stdout
              - Filebeat autodiscover Docker containers → ES index chat-logs-*
              - Kibana: Discover panel + Dashboard error rate

☐ 9.4  Alerting Rules (Prometheus Alertmanager)
       - Kafka consumer lag > 10,000
       - Redis memory > 80%
       - WebSocket connections spike
       - Error rate > 1%

☐ 9.5  Disaster Recovery Plan
       - RTO: 15 phút, RPO: 5 phút
       - Runbook cho các sự cố thường gặp

☐ 9.6  Capacity Planning Document
       - 1M users: 20 app instances, 3 Kafka brokers,
         6-node Redis cluster, 3-node MongoDB replica set

☐ 9.7  Production Deployment Checklist
       - Environment variables review
       - Smoke test suite
       - Rollback procedure
```
